HOW THE FIX WORKS
─────────────────────────────────────────────────────────────────────────────

Before Migration Start:
  1. User generates key on Etch
  2. User pastes key on Bricks → stored in JavaScript
  3. User clicks "Start Migration" → sends to backend
  4. Backend VALIDATES but DOESN'T SAVE ← BUG
  5. Frontend polls get_progress()
  6. get_progress() looks in Settings → FINDS NOTHING → ERROR

After Migration Start (With Fix):
  1. User generates key on Etch
  2. User pastes key on Bricks → stored in JavaScript
  3. User clicks "Start Migration" → sends to backend
  4. Backend VALIDATES and SAVES to Settings ← FIX
  5. Frontend polls get_progress()
  6. get_progress() looks in Settings → FINDS KEY → SUCCESS
  7. Extracts target_url from JWT payload
  8. Migration continues normally

KEY ARCHITECTURAL POINTS
─────────────────────────────────────────────────────────────────────────────

1. JWT is single source of truth for target_url
   - Settings only stores: migration_key (the JWT token)
   - Never stores: target_url separately
   - URL extracted from JWT payload on each request

2. Key must be persisted after start_migration()
   - Frontend only sends it once (at start)
   - Backend must save for later retrieval
   - All subsequent calls need it for JWT decoding

3. No Settings['target_url'] pattern
   - This creates sync issues and bugs
   - JWT payload is the source
   - Extract dynamically, don't persist separately

ISSUE FIXED
──────────────────────────────────────────────────────────────────────────────

User reported: Migration failed on Bricks with "configuration_incomplete" error

Root Cause: Migration key was received but never saved to WordPress Settings
             → get_progress() couldn't find it
             → Couldn't extract target_url from JWT
             → Migration blocked with error

FIX APPLIED
──────────────────────────────────────────────────────────────────────────────

File: includes/controllers/class-migration-controller.php
Lines: 53-57 (added after migration_key validation)

Code:
    // Save migration_key to Settings for later retrieval by get_progress().
    // This ensures the key is available for all subsequent API calls 
    // without re-sending it in every request.
    $settings                  = get_option( 'efs_settings', array() );
    $settings['migration_key'] = $migration_key;
    update_option( 'efs_settings', $settings );

Why This Works:
  ✓ Persists the migration_key for retrieval by get_progress()
  ✓ Allows JWT decoding to extract target_url
  ✓ Enables subsequent API calls to Etch site
  ✓ Minimal change - only 3 lines
  ✓ No breaking changes
  ✓ Backward compatible

KEY LESSONS
──────────────────────────────────────────────────────────────────────────────

Architecture Pattern to Remember:
  • Frontend sends data to backend
  • Backend MUST persist it for later use
  • Never assume frontend will resend the same data
  • Build with stateless HTTP in mind (each request is independent)

JWT Best Practice:
  • Token contains all needed info (target_url, source, expiry)
  • Store token in Settings, not extracted fields
  • Extract fields from token on demand
  • Single source of truth prevents sync bugs

Code Quality:
  • Always test full workflow (call A → call B using A's result)
  • Document why (architecture) not just what (code)
  • Make minimal changes to fix root causes
  • Maintain backward compatibility
